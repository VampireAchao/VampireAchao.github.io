Hallmark：一项专门用来拒绝“AI 味”的设计技能

当越来越多的页面、落地页和产品界面开始由 AI 参与生成，一个新问题也越来越明显：它们往往“看起来都像会说话的模板”。结构熟悉，措辞顺手，排版工整，可一眼望过去，灵魂像被同一种默认值抹平了。

Hallmark 就是冲着这件事来的。

根据仓库 description，它是一个面向 Claude Code、Cursor 和 Codex 的 anti-AI-slop design skill。README 则把这件事说得更有锋芒：这是一个拒绝看起来像 AI 生成物的设计技能。

项目地址：https://github.com/Nutlope/hallmark

它不是在帮 AI 生成页面，而是在逼 AI 别再偷懒

README 开头的那句自我介绍非常有辨识度：

A design skill for Claude Code, Cursor, and Codex that refuses to look AI-generated.

这句话的重点，不只是“设计技能”，而是 refuses。
Hallmark 给人的第一感觉，不像一个热情帮忙出稿的助手，更像一个坐在旁边、对俗套默认值极度不耐烦的创意总监。它不是单纯提高生成速度，而是在盯着生成结果里最容易滑向平庸的那一层。

README 对它的工作方式也描述得很具体：

  • 为 brief 选择一种 macrostructure
  • 套上一种主题
  • 运行 57 个 slop-test gates
  • 在输出前进行一次自我批评
  • 拒绝大模型那些“高分布默认值”

这几句话拼在一起，就能看出 Hallmark 的方法论很明确：
它不迷信“让模型自由发挥”，而是给生成过程立下了一套审美纪律。

它有四个动词，而不是一种单一输出方式

Hallmark 并不是只能“帮你做新页面”。README 用“四个 verbs”来组织它的能力，这个设计很漂亮，因为它把设计工作拆成了几种完全不同的创作动作。

默认动词

默认模式用来 Build new UI。
它会为新 brief 选择 macrostructure，套用规则集，并在交付前跑一遍 slop test。

也就是说，默认不是无条件吐页面，而是先挑结构、再按规则约束，再交付。

hallmark audit <target>

这个动词用来 审视已有代码。
README 说得很明确：它会根据 anti-patterns 去打分，输出 punch list，但不做修改。

这很像一个只看病、不动刀的设计评审模式。

hallmark redesign <target>

这个动词最有攻击性。
它会 扔掉原有结构，保留 copy、信息架构和品牌，再用不同的 fingerprint 重新搭建。

这不是修边角，而是重构骨架。

hallmark study <screenshot | URL>

这是一个很有意思的能力。
它会从一个你欣赏的设计中提取 DNA，包括：

  • macrostructure
  • type-pairing
  • colour anchor

同时，README 也明确强调：它会拒绝 pixel-clones 和 paid templates。

这意味着它想做的是“学习语法”，不是“临摹成品”。

它最迷人的地方，是不同 brief 会长出不同形状

README 有一节标题写得非常准确：

Different briefs, different shapes

这几乎就是 Hallmark 的核心审美宣言。

它展示了多组示例页面，每一个都来自不同 brief，而技能会根据 brief 去决定：

  • 主题
  • 结构
  • 具体 craft

而不是从某个固定模板里套出来。

README 甚至直接强调：
The skill picks the theme, structure, and craft to fit each one, not from a template.

这点特别重要。
因为很多所谓“AI 设计能力”表面上看提供了丰富视觉样式，实际只是把同一个布局骨架反复换皮。Hallmark 显然不满意这种做法。它要的不是“这一页是蓝色，那一页是米色”，而是从结构层就开始分化。

每个页面都是自包含的，而且带着自己的“结构印章”

README 提到，这些生成出来的页面是：

  • self-contained HTML + CSS
  • 并且在 CSS comment 中带有它的 macrostructure stamp

这件事很有意思。
它意味着 Hallmark 不只是产出视觉结果，还会把“这页属于哪一种结构指纹”这件事写进作品本身。

这有点像一位设计师在画布背面留下构图注释：表面上你看到的是成品，背后则清楚记着它属于哪一种设计方法。

“Custom” 分支，是 Hallmark 很关键的一步

README 中有一节专门叫 Custom NEW。

它说明，当一个 brief 携带的创意意图超出了主题目录的适配范围时，Hallmark 会切换到 Custom 路线，从头为这个页面设计：

  • 定制调色板
  • 定制字体搭配
  • 定制布局

但它也强调：
即使走到 Custom 路线，仍然要经过同样的 57 个 slop-test gates。

这意味着 “Custom” 不是放飞，而是 在纪律框架内做更个性化的设计决定。

README 还说得很清楚：

  • 这是一个安静分支
  • 普通 brief 不会自动进入它
  • 具体协议写在 custom-theme.md

换句话说，Custom 并不是 Hallmark 默认拿来卖弄创意的入口，而是在确实遇到目录主题装不下的 brief 时，才被唤起的一条更细致路线。

site/_tests 里的测试说明，把它的设计方法暴露得很彻底

仓库中的 site/_tests/README.md 很值得读，因为它几乎把 Hallmark 的实验精神端到了桌面上。

它描述了八个 landing page 测试，这些测试的目的不是“证明我们能做页面”，而是验证更具体的事情，例如：

  1. 不同 brief 是否真的不会默认成长得一样
  2. 技能是否会在不该加装饰的时候克制自己
  3. 当用户跳过设计上下文时，技能是否会清楚披露自己的推断
  4. 生成文案时是否会滑向模板化口吻

这几项观察点都很尖锐。
它们说明 Hallmark 不只是追求“页面好不好看”，而是在系统性对抗设计生成里最常见的几种退化：

  • 结构重复
  • 装饰冲动
  • 默认语气模板化
  • 上下文处理偷懒

site/_tests/README.md 还反复强调一个很有代表性的目标：
No macro repeats. No theme repeats.

这几乎可以看成 Hallmark 的创作纪律之一：相邻输出之间必须有可感知的结构差异，而不是只换色。

它会认真复盘自己的毛病，而且语气毫不客气

site/_tests/README.md 最精彩的一部分，可能是 Findings。

它一方面总结什么地方做得好，比如：

  • 六个不同 brief 确实产出了六个不同结构页面
  • 某些页面正确地保持了 typography-only，没有强行加 enrichment
  • 某些手工 CSS art 不是靠 Lottie 糊弄
  • skipped-context flow 运行正常
  • 不同测试中的 type pairing 保持了差异

另一方面，它也非常具体地承认 friction points，比如：

  • domain → trio table 覆盖了大多数情况，但不是全部
  • theme picking 仍然有“凭感觉”的部分
  • 文案有时会滑向类似 “Built for the modern team” 这种模板味
  • hero-enrichment.md 的说明对某些情境还不够具体
  • within-archetype 的变化旋钮虽然有，但自己有时还是会倾向默认值
  • 免费字体替代在某些气质上仍然有遗憾

这种复盘方式非常迷人。
它不像一个只会展示“成就”的营销 README，而像一个真正的设计系统在做自我校验：哪里做对了，哪里还不够，下一轮该补什么洞，全都摊开说。

它还有一份明确的改进清单，而且一项项在推进

在 site/_tests/README.md 里，Hallmark 不只是指出问题，还给出了分层的 improvement list：

  • Tier 1 — Real holes
  • Tier 2 — Quality polish
  • Tier 3 — Long bet

更有意思的是，后文还记录了哪些改进已经在后续版本中完成，例如：

  • per-archetype copy fixtures
  • theme-diversification rule
  • free-vs-paid font trade-off
  • worked examples per enrichment archetype
  • non-LLM voice samples
  • project memory .hallmark/log.json
  • visual thumbnails on the example gallery
  • new slop-test gates
  • SaaS-focused tests
  • default-on microinteractions rule

从这些内容能看出，Hallmark 不是把“anti-slop”当成一句态度口号，而是在不断往规则、测试、示例和流程里补充更精细的约束。

“Custom worked examples” 进一步说明：它的个性化不是乱飘，而是有守卫的

site/_tests/custom/README.md 对 Custom 路线做了很好的补充。

它列出了几组 custom worked examples，并说明 Custom 适用于那种：

  • 目录主题无法承载的品牌气质
  • 需要单独调色与字体搭配
  • 有明确 vibe 或色彩锚点的 brief

例如文档中提到的案例，会围绕：

  • terracotta
  • sea-blue
  • dusty-pink

这样的 anchor 去形成不同方向的视觉选择。

但同一份 README 也强调了防止过度发明的 guard：

  1. Custom 只在用户确实表达独特创意意图时才触发
  2. 只问一个问题：vibe 加可选 anchor colour
  3. 现有规则仍然全部适用
  4. Step 5 preview 会先把 palette 和 pairing 明文展示出来
  5. 自定义路线同样纳入 .hallmark/log.json 做轮换记录

这一整套 guard 很重要。
它保证了 Hallmark 的“个性化”不是漫无边界地即兴发挥，而是仍然受到一整套结构纪律和审美测试的管束。

安装方式非常简单，说明它本质上是“技能”，不是传统框架

README 的安装部分很简洁：

并说明可以随时重新执行以更新。

此外，也可以手动复制 SKILL.md 与 references/ 到不同环境里：

  • Claude Code
  • Cursor
  • Codex

这些安装路径本身也再次说明了 Hallmark 的产品形态：
它不是一个重型设计系统框架，而更像一份可被不同 AI coding 环境加载的设计技能集。

README 还明确指出：

  • 规则集在 SKILL.md
  • 参考资料在 references/
  • worked examples 在 docs/recipes.md 和 docs/study-examples.md

它的核心不是运行某段应用代码，而是把一套设计判断准则、生成流程和反模板纪律，组织成可复用的 skill。

它的“反 AI 味”不是粗暴反对 AI，而是反对懒惰默认值

有些项目会把“anti-AI”理解成反技术姿态，但 Hallmark 不是。
它并没有拒绝 AI 生成，相反，它完全是为 AI 工作流准备的。它真正反对的，是那种：

  • 默认结构反复重演
  • 视觉指纹相互重叠
  • 文案滑向模板汤
  • 一切都像被同一个大模型平均值熨平

所以 Hallmark 的敌人不是 AI，而是 AI 的分布式惯性。
它的工作，不是把 AI 推开，而是拽住 AI 的衣领，对它说：别再走最省力、最像别人、最容易一眼看穿的那条路。

从仓库信息看，它的身份也很鲜明

根据仓库 metadata：

  • 仓库名为 Nutlope/hallmark
  • 默认分支是 main
  • 主要语言是 CSS
  • License 为 MIT
  • description 是 Anti-AI-slop design skill for Claude Code, Cursor, and Codex

这组信息和 README 呈现出来的整体气质完全一致：
一个围绕设计输出、结构差异、视觉规则与前端呈现组织起来的技能仓库，不是传统意义上的应用产品，而是一个有强烈审美立场的生成规则系统。

如果把 Hallmark 拟人化，它像一个对“模板味”过敏的设计总监

你给它 brief，它不会立刻点头说“好的我来帮你做个现代感首页”。
它会先皱一下眉，翻一遍你的上下文，挑一个不重复的结构，避开最近刚用过的主题，检查文案有没有滑向陈词滥调，再过一遍几十道 slop-test gate。
如果你要求它研究一个喜欢的页面，它也不会照着描，而是只提炼 DNA，不偷皮相。
如果你的 brief 太有个性，现成主题兜不住，它才会悄悄走进 Custom 分支，给你配一套单独的颜色、字体和布局，但仍旧把纪律带在身上。

结语

Hallmark 最有意思的地方，不是“它能生成漂亮页面”，而是：它试图系统性地对抗 AI 生成设计中最容易出现的俗套、重复和模板化。 README 里的四个 verbs、macrostructure 选择、主题轮换、57 个 slop-test gates、预输出自我批评、Custom 分支、测试集与改进清单，共同勾勒出一套非常鲜明的方法论。

它不像一个热衷炫技的生成器，更像一套有脾气、有纪律、会自我审查的设计技能。
而在一个“做得快”越来越容易、“做得不像默认 AI 输出”反而越来越难的时代，Hallmark 恰好把刀锋磨在了后者身上。
